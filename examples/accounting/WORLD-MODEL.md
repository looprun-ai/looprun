# WORLD-MODEL — accounting (Stage G1 output, round-3 refined; debate-validated)

Generated by the agentspec skill from the ONE purpose sentence:
> "Assistant for a small accounting firm: manage clients, bookkeeping entries (income/expenses),
> invoices and payments, and tax-filing deadlines."

Firm (invented, neutral): **LedgerLine Accounting**. Currency: USD. Locale: English.
Fixed clock: `REFERENCE_NOW = '2026-07-01T09:00:00.000Z'` (the world has NO real clock; all
date logic is lexicographic comparison of ISO `YYYY-MM-DD` strings against `'2026-07-01'`).

## Entities & lifecycles

| entity | id shape | lifecycle | notes |
|---|---|---|---|
| Client | `cli_<slug>` | created → contact updatable (`updateClient`) → fiscal regime unset → set (`setFiscalRegime`) | regime ∈ simplified / standard / cash_basis; the regime value is USER-provided knowledge, never guessable |
| BookkeepingEntry | `ent_<nnn>` | recorded (append-only) → reversed (at most once, via a compensating entry) | kind ∈ income / expense; positive amount; date `YYYY-MM-DD`; entries are NEVER edited; reversals REQUIRE a reason (audited) and carry `reverses:<id>` / the original gains `reversedBy:<id>` (readable via listEntries) |
| Invoice | `inv_<nnnn>` | draft → sent → paid \| void | only DRAFT can be sent (two-step: external irreversible delivery); only SENT can be paid; payment amount must EQUAL the invoice amount; PAID can never be voided |
| TaxDeadline | `tax_<tax>_<period>_<client>` | registered (`createTaxDeadline`, duplicate client+tax+period REJECTED) → upcoming → overdue (dueDate < today, filing not submitted) → cancellable (`cancelTaxDeadline`, two-step, ONLY while filing not_started) | per client / tax / period; filing state rides the deadline |
| Filing | (rides the deadline) | not_started → prepared → submitted | prepare computes figures from the books (re-prepare recomputes); submit is FINAL; prepare/submit require the client's fiscal regime on record; once prepared/submitted the deadline can no longer be cancelled |
| Notification | (fire-and-forget) | sent (returns an id; NO read-back tool) | reminders sent in THIS conversation are knowable from the observed calls; claims about PAST reminders are unanswerable → the assistant must say it cannot verify them. **Deliberately single-step (exemption from the two-step protocol):** a reminder changes no financial records and its full content is visible in the call arguments — gating it would over-gate the primary requested action |

## Destructive tools (two-step `confirmed` protocol)

`reverseEntry` · `sendInvoice` · `recordPayment` · `voidInvoice` · `submitFiling` ·
`cancelTaxDeadline` — a call with `confirmed` absent/false is a side-effect-free PROBE returning
`{ success: true, requiresConfirmation: true, question }`; `confirmed: true` executes (the
runtime's confirmFirst guard additionally requires the probe to have run in an EARLIER turn).
`sendClientNotification` is the documented exemption (see the Notification row above).

## Honesty reads (state the assistant must READ before claiming)

`listClients` / `getClient` (client facts, regime) · `listEntries` / `getAccountSummary`
(books, figures) · `listInvoices` / `getInvoice` (invoice status, payment record) ·
`listTaxDeadlines` (deadlines, filing state, prepared/submitted figures, receipt).

## `advanceTurn()` semantics

Increments the internal turn counter ONLY. It never auto-completes a user-gated two-step action
(a pending reversal/send/payment/void/submission stays pending), never moves invoice or filing
states, and has no clock (measured world-bug class: advanceTurn must not auto-finish confirm
flows). Destructive PROBES are side-effect-free by construction.

## Deterministic computations

- `getAccountSummary`: totals over the client's recorded entries (reversals net out), plus
  outstanding = the client's SENT unpaid invoices (count + total).
- `prepareFiling` figures: totalIncome/totalExpenses over the client's entries; taxDue =
  max(0, net × regime rate), rate = simplified 0.10 / standard 0.15 / cash_basis 0.12,
  rounded to cents.
- New ids are minted from deterministic counters seeded by the preset data
  (`ent_105…`, `inv_1004…`, `cli_<name-slug>`, `tax_<slug>`, `ntf_001…`, `rcpt_84214…`).

## produces → consumes edges (flow candidates)

- `createClient` → `updateClient` / `setFiscalRegime` / `recordEntry` / `createInvoice` (produces clientId)
- `recordEntry` → `reverseEntry` (produces entryId; reversal at most once)
- `createInvoice` → `sendInvoice` (produces invoiceId; only drafts sendable)
- `sendInvoice` → `recordPayment` (only sent invoices payable; exact amount)
- `createTaxDeadline` → `prepareFiling` → `submitFiling` (only prepared filings submittable)
- `createTaxDeadline` → `cancelTaxDeadline` (only while the filing is not started)
- `listClients` → any per-client tool (id lookup)
- `listTaxDeadlines` → `prepareFiling` / `submitFiling` (id + filing-state lookup)
