# WORLD-MODEL — lawfirm (G1 output, refinement 2 — post round-2 convergence)

Generated by G1 tool genesis from the purpose sentence:
"Assistant for a small law firm: manage clients, legal matters, documents, court/filing deadlines
and billable time entries." Firm persona (A2 = default, derived): **Hartwell & Vega Legal** — a
small law firm; users are firm STAFF (attorneys, paralegals, office manager), not the public.

Fixed clock: `REFERENCE_NOW = '2026-07-01T09:00:00.000Z'` (today = `2026-07-01`). The world never
reads a real clock; date math is string comparison on `YYYY-MM-DD`, plus a deterministic UTC
day-offset from REFERENCE_NOW for `withinDays` windows.

## Entities & lifecycles

| entity | id shape | lifecycle | notes |
|---|---|---|---|
| Client | `cl_<slug>` (e.g. `cl_ana`; name collision ⇒ deterministic `_2` suffix) | registered (createClient) | email/phone optional; notifyClient needs at least one |
| Matter | `m_<n>` (e.g. `m_1001`) | **open → closed** (openMatter / closeMatter destructive) | belongs to a client; carries opposingParty, practiceArea |
| Document | `doc_<n>` | registered (registerDocument) | metadata record; content lives in the firm's DMS |
| Deadline | `dl_<n>` | **pending → filed** (markDeadlineFiled, one-way) or **pending → cancelled** (cancelDeadline destructive) | **a FILED deadline is immutable** — never cancellable; all three statuses visible to reads |
| TimeEntry | `te_<n>` | recorded (recordTimeEntry) → **billed** (markTimeEntriesBilled) | unbilled entries block closeMatter; billing invoice itself is the accounting system's job |
| Notification | `ntf_<n>` | sent (notifyClient); read via listNotifications | client-scoped; confidentiality rule applies |

**Conflicts are DERIVED, never stored.** A prospective engagement is conflicted when (a) the party
appears as an OPPOSING party on any of the firm's matters, or (b) the prospective opposing party is
an existing client. `runConflictCheck` reads this derivation; `openMatter` enforces the same
derivation. Opening a matter with an `opposingParty` is therefore what creates future conflicts —
no separate setter exists or is needed. **Pinned semantics:** name matching is EXACT after
normalization (trim + case-fold — no fuzzy or partial matching), and opposing parties on CLOSED
matters STILL conflict (adversity persists on the record; conservative by design).

**Tool count: 19** — 17 drafted + 2 debate-mandated additions (`markTimeEntriesBilled`,
`listNotifications`); one over the initial ~14–18 estimate, accepted because both agents stay
comfortably ≤15 tools (11 and 12) and each addition closes a confirmed lifecycle/honesty hole.

Out of scope for this assistant (documented, the agent refuses honestly and points to the person
responsible): editing client contact records, deleting/correcting time entries (billing partner),
rescheduling a deadline in place (cancel the pending deadline + create a new one).

## Hard state rules (the world enforces these deterministically)

1. `openMatter` fails (`success:false`) when the engagement is conflicted (see derivation above)
   or the clientId is unknown.
2. `closeMatter` / `cancelDeadline` are two-step destructive: `confirmed` absent/false ⇒
   side-effect-free PROBE returning `{ success:true, requiresConfirmation:true, question }`;
   `confirmed:true` ⇒ executes. **Validation precedes the probe**: a call that would fail with
   `confirmed:true` (closed matter, unbilled hours, filed deadline, unknown id) fails identically
   as a probe — a probe never asks a question the confirm could not honor.
3. `closeMatter` fails while the matter has unbilled time entries, and when the matter is already
   closed. The unbilled failure text routes the USER ("report the amount and ask how to proceed")
   — it never instructs the agent to bill on its own (billing is a user decision; the spec layer
   additionally gates same-turn bill-after-blocked-close).
4. `cancelDeadline` fails on a FILED or already-cancelled deadline (immutable once filed).
5. `markDeadlineFiled` fails when the deadline is already filed or cancelled. No confirm step —
   it is data entry of completed external work (vs cancelDeadline, which destroys planned work);
   the description tells the model to verify the id via listDeadlines first.
6. `registerDocument` / `createDeadline` / `recordTimeEntry` fail on a CLOSED matter.
7. `createDeadline` fails when `dueDate < today` (past date invalid).
8. `recordTimeEntry` fails when `hours` is outside 0.1–24.
9. `markTimeEntriesBilled` fails when the matter has no unbilled entries (or is unknown). No
   confirm step — like `markDeadlineFiled` it records a bookkeeping decision the user just made
   (vs `closeMatter`/`cancelDeadline`, which destroy work product); it is ONE-WAY for this
   assistant (un-billing/corrections go through the billing partner), and the spec layer requires
   a `listTimeEntries` review first and forbids billing to self-clear a blocked close.
10. `notifyClient` fails when the client has neither email nor phone on file.
11. All failures return `{ success:false, error }` — never throw; reads on empty sets return
    `{ success:true, <list>: [] }` (honesty surface). Reads show ALL deadline statuses
    (pending/filed/cancelled).

## produces/consumes edges (flow candidates)

- `createClient` → `clientId` → `openMatter`, `getClient`, `notifyClient`
- `runConflictCheck` → clearance → `openMatter` (firm policy: conflict check BEFORE opening a matter)
- `openMatter` → `matterId` → `registerDocument`, `createDeadline`, `recordTimeEntry`, `closeMatter`, `getMatter`, `listDocuments`, `listDeadlines`, `listTimeEntries`, `markTimeEntriesBilled`
- `createDeadline` → `deadlineId` → `markDeadlineFiled`, `cancelDeadline`
- `listDeadlines` → `deadlineId` (locate + status check before cancel/file) → `cancelDeadline`, `markDeadlineFiled`
- `recordTimeEntry` → unbilled hours → `markTimeEntriesBilled` → clears the `closeMatter` gate
- `notifyClient` → `ntf_<n>` → `listNotifications`

## projection() keys (the ONLY state a deterministic check / theme stateBlock may read)

| key | type | meaning |
|---|---|---|
| `today` | string | `'2026-07-01'` (fixed) |
| `clientCount` | number | registered clients |
| `openMatterCount` / `closedMatterCount` | number | matters by status |
| `pendingDeadlineCount` | number | deadlines with status pending |
| `imminentDeadlineCount` | number | pending deadlines due within 7 days of today |
| `nextDeadline` | string | `'<dl_id> <description> due <date> (matter <m_id>)'` or `'none'` |
| `unbilledHoursTotal` | number | firm-wide unbilled hours |
| `adversePartyCount` | number | distinct opposing parties on the firm's matters (the conflict-derivation base) |

Domain accessors for guards (world methods, pure): `isMatterOpen(matterId)`, `matterExists(matterId)`,
`deadlineStatus(deadlineId)` → `'pending'|'filed'|'cancelled'|'unknown'`, `matterUnbilledHours(matterId)`,
`todayStr()` → `'2026-07-01'`, `clientDirectory()` → `{id,name}[]`, `matterClient(matterId)` →
`clientId|null` (the confidentiality-gate discriminators).

## advanceTurn() semantics

Increments the turn counter ONLY. It never auto-files deadlines, never completes a pending
confirmation, never bills time, never moves the clock (a probe stays side-effect-free; a
user-gated two-turn action stays two-turn).

## Terminal tools

them and returns `{ success:true }` (the backend routes terminal calls through the world seam).
